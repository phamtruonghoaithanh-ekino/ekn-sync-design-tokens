# EKN Tokens Drupal theme

A Drupal 10/11 custom theme that consumes complete, theme-specific CSS files generated by Style Dictionary.

> This repository is a Drupal theme, not a standalone application. It must be installed inside a Drupal project at `web/themes/custom/ekn_tokens`.

## Requirements

- Drupal 10.3 or later, or Drupal 11
- PHP and extensions required by the selected Drupal version
- Composer 2
- Drush installed in the Drupal project
- A working Drupal database and web server

Drupal 10.3 or later is required because Single Directory Components are part of Drupal's core render system from that release onward.

## Optional: create a fresh local Drupal site with DDEV

Skip this section if you already have a working Drupal project. It requires Docker and [DDEV](https://ddev.readthedocs.io/en/stable/).

Create and install a Drupal 11 site:

```sh
mkdir my-drupal-site
cd my-drupal-site
ddev config --project-type=drupal11 --docroot=web
ddev start
ddev composer create-project "drupal/recommended-project:^11"
ddev composer require drush/drush
ddev drush site:install --account-name=admin --account-pass=admin -y
ddev launch
```

Then install this theme at `web/themes/custom/ekn_tokens` using Step 1 below.

When using DDEV, replace commands beginning with `vendor/bin/drush` in this guide with `ddev drush`. For example:

```sh
ddev drush theme:enable ekn_tokens -y
ddev drush config:set system.theme default ekn_tokens -y
ddev drush cache:rebuild
```

## Optional: run a local Drupal site with Docker Compose

This repository includes a Drupal 11, Apache, MariaDB, and Drush development
stack. The current checkout is mounted directly as the `ekn_tokens` theme, so
theme file changes are visible after a Drupal cache rebuild.

Docker Desktop (or Docker Engine with Compose v2) is required.

Build and start the containers:

```sh
docker compose up -d --build
```

Install Drupal and enable this theme:

```sh
docker compose exec drupal vendor/bin/drush site:install standard \
  --db-url=mysql://drupal:drupal@db/drupal \
  --site-name="EKN Tokens" \
  --account-name=admin \
  --account-pass=admin \
  -y
docker compose exec drupal vendor/bin/drush theme:enable ekn_tokens -y
docker compose exec drupal vendor/bin/drush config:set system.theme default ekn_tokens -y
docker compose exec drupal vendor/bin/drush cache:rebuild
```

Open <http://localhost:8080> and sign in with `admin` / `admin`. These
credentials are intentionally development-only.

Useful commands:

```sh
# View container status and logs.
docker compose ps
docker compose logs -f drupal

# Rebuild caches after changing Twig, YAML, or PHP files.
docker compose exec drupal vendor/bin/drush cache:rebuild

# Stop the site while preserving its database and uploaded files.
docker compose down

# Remove the local site and database completely.
docker compose down -v
```

## 1. Install the theme in an existing Drupal project

Run these commands from the Drupal project root—the directory containing `composer.json` and `web/`.

Create the custom-theme directory if it does not exist:

```sh
mkdir -p web/themes/custom
```

Then use one of these installation methods.

### Clone the repository

Replace `<repository-url>` with this repository's Git URL:

```sh
git clone <repository-url> web/themes/custom/ekn_tokens
```

### Link a local checkout

This is useful when developing the theme from its current checkout:

```sh
ln -s /absolute/path/to/ekn-sync-design-tokens web/themes/custom/ekn_tokens
```

The destination directory must be named `ekn_tokens`. Drupal derives PHP hooks, component IDs, configuration names, and library names from this machine name.

## 2. Install Drush if needed

From the Drupal project root:

```sh
composer require drush/drush
```

Confirm that Drush can connect to Drupal:

```sh
vendor/bin/drush status
```

## 3. Enable the theme

```sh
vendor/bin/drush theme:enable ekn_tokens -y
vendor/bin/drush config:set system.theme default ekn_tokens -y
vendor/bin/drush cache:rebuild
```

Confirm the active default theme:

```sh
vendor/bin/drush config:get system.theme default
```

Expected value:

```text
ekn_tokens
```

You can also enable and select the theme at **Appearance** (`/admin/appearance`).

## 4. Select the generated token theme

Open **Appearance → Settings → EKN Tokens** and select one of:

- React Light
- React Dark
- React Brand A

Save the configuration and rebuild caches:

```sh
vendor/bin/drush cache:rebuild
```

The same setting can be changed from the command line:

```sh
vendor/bin/drush config:set ekn_tokens.settings active_token_theme react_dark -y
vendor/bin/drush cache:rebuild
```

Valid values are:

```text
react_light
react_dark
react_brand_a
```

Confirm the stored value:

```sh
vendor/bin/drush config:get ekn_tokens.settings active_token_theme
```

## 5. Verify that only one theme CSS file is loaded

Open any frontend page, then inspect the `<html>` element in browser developer tools.

Expected attributes include one of:

```html
<html data-theme="react-light">
<html data-theme="react-dark">
<html data-theme="react-brand-a">
```

Open the browser's **Network** panel, filter by `vars-React`, and reload the page. Exactly one generated token file must be loaded:

```text
Light   → vars-React Light.css
Dark    → vars-React Dark.css
Brand A → vars-React Brand A.css
```

If multiple generated token files are present, inspect custom modules or additional theme libraries. This theme attaches only one allow-listed token library in `ekn_tokens_preprocess_html()`.

## 6. Render and test the SDC components

The theme provides these component IDs:

```text
ekn_tokens:button
ekn_tokens:card
```

Use the button from a Drupal Twig template:

```twig
{{ include('ekn_tokens:button', {
  label: 'Continue',
  variant: 'primary',
  type: 'button'
}, with_context = false) }}
```

Valid button variants are `primary` and `secondary`. Valid types are `button`, `submit`, and `reset`.

Use the card and its slots:

```twig
{% embed 'ekn_tokens:card' with {
  title: 'Token-based card',
  heading_level: 'h2'
} only %}
  {% block content %}
    <p>This content is rendered inside the card component.</p>
  {% endblock %}

  {% block actions %}
    {{ include('ekn_tokens:button', {
      label: 'Continue',
      variant: 'primary'
    }, with_context = false) }}
  {% endblock %}
{% endembed %}
```

After adding or changing a Twig template, rebuild Drupal caches:

```sh
vendor/bin/drush cache:rebuild
```

Expected behavior:

- Component CSS is discovered automatically by Drupal's SDC system.
- Button and card colors change when the active token theme changes.
- Component spacing, borders, and radii come from CSS custom properties.
- Native buttons work with the keyboard and expose disabled state correctly.

SDCs are reusable from Twig immediately. Exposing them as editor-selectable Layout Builder blocks requires a project-specific block plugin or an SDC/Layout Builder integration module.

## 7. Run static checks

Run from the theme directory.

Validate PHP syntax:

```sh
php -l ekn_tokens.theme
php -l theme-settings.php
```

Check patch whitespace:

```sh
git diff --check
```

Confirm that all three generated files exist:

```sh
find css/tokens -maxdepth 1 -name 'vars-React*.css' -print
```

Expected files:

```text
css/tokens/vars-React Light.css
css/tokens/vars-React Dark.css
css/tokens/vars-React Brand A.css
```

The Drupal integration check is a successful cache rebuild after enabling the theme:

```sh
vendor/bin/drush cache:rebuild
```

This discovers invalid theme YAML, component metadata, and library definitions in the running Drupal environment. Twig syntax is exercised when the relevant template or component is rendered.

## 8. Test accessibility manually

Repeat these checks for Light, Dark, and Brand A:

1. Navigate all interactive elements using only `Tab` and `Shift+Tab`.
2. Confirm every focused link, button, input, and control has a visible focus indicator.
3. Confirm text and controls remain usable at 200% browser zoom.
4. Check the page at mobile, tablet, and desktop widths.
5. Run axe DevTools or another WCAG 2.2 AA checker.
6. Check color contrast for normal text, large text, controls, and focus indicators.
7. Confirm buttons have meaningful visible labels.
8. Confirm heading levels passed to the card match the page hierarchy.

Step 4 implements accessible component behavior. A passing audit still requires browser-level WCAG 2.2 validation against the complete Drupal page.

## 9. Update generated token files

Generated files are stored in `css/tokens/`:

- `vars-React Light.css`
- `vars-React Dark.css`
- `vars-React Brand A.css`

Do not edit these files manually. Replace them with new Style Dictionary output while preserving their filenames and CSS custom-property contract, then run:

```sh
vendor/bin/drush cache:rebuild
git diff --check
```

Review the generated diff and repeat the theme and accessibility checks above.

## Troubleshooting

### Drupal cannot find the theme

Confirm the path and machine name:

```text
web/themes/custom/ekn_tokens/ekn_tokens.info.yml
```

Then rebuild caches:

```sh
vendor/bin/drush cache:rebuild
```

### Theme changes do not appear

Rebuild caches and temporarily disable CSS aggregation at **Configuration → Development → Performance**.

```sh
vendor/bin/drush cache:rebuild
```

### The selected token theme is invalid

The preprocess hook safely falls back to `react_light`. Reset the configuration:

```sh
vendor/bin/drush config:set ekn_tokens.settings active_token_theme react_light -y
vendor/bin/drush cache:rebuild
```

### Component changes do not appear

Confirm the component files use matching names:

```text
components/button/button.component.yml
components/button/button.twig
components/button/button.css
```

Then rebuild caches. Drupal automatically attaches a component's CSS when that component is rendered.
